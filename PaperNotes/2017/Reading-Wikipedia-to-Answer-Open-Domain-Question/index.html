Reading Wikipedia to Answer Open-Domain Question

Posted on 2018-02-13 | In Paper Notes | | Views:

Paper Link

Abstract

  • 只使用 Wiki 當作資源找出趣聞問題的解答
  • Machine Reading at Scale (MRS)，有 document retrieval + machine comprehension of text 的問題要解決
  • System: Search Component + RNN
    • Bigram hashing
    • TF-IDF matching
    • Multi-layer RNN

Introduction

  • 一般 knowledge bases(KBs)：Freebase, DBPedia
  • 使用 Wiki 作文本導致 QA任務 需要結合 large-scale open-domain QA 和 machine comprehension

  • Machine Reading at Scale:

    • 抓出相關文章
    • 掃描段落找出答案

  • 只使用內文，不使用 wiki 的 internal graph structure

    • 可以套用此系統到其他文本

  • Large-scale QA system: IBM’s DeepQA

    • 依賴很多資源
    • information redundancy

  • Machine Comprehension: SQuAD, CNN/Daily Mail

    • 大多都是以找出一小段相關 text 給 model，但是對 open-domain QA 不切實際
    • 因此， KBs or Information Retrieval 應該要有 search component

  • 使用很多 dataset 來評估 MRS

  • Improve performance by the use of multitask learning and distant supervision

    ditant supervision: 使用已從在的資料庫(Freebase or a domain-specific DB)來搜集有關聯的配對，然後再用這些配對產生 training data。例如：Freebase 有 Barack Obama & Michelle Obama 結婚，拿這個事實去標籤同時包含這兩位的句子，如此就能產生大量的資料（也有可能 noisy）。

Related Work

  • Motivation:
    • KBs 限制: 不完整、固定的資料格式
    • 近期 deep learning 發展，attention-based, memory-augmented neural networks
  • 提及其他使用 wiki 的研究 blablabla…
  • 只考慮 text comprehension
  • 為了強調 machine reading at scale 的任務，只使用 wiki 當作唯一資源

DrQA for MRS

  • Doc Retrieval + Doc Reader

  • Doc Retrieval

    • 使用有效率的方式（非機器學習）
    • simple inverted index lookup(followed by term vector model scoring???) > built-in ElasticSearch based Wiki Search API
    • 使用 TF-IDF weighted bag-of-word vectors 比較文章和問題
    • 透過 local word order with n-gram features 改良系統，best -> bigram counts -> 使用 hashing 保持速度和記憶體使用率 -> map bigrams to $2^{24}$ bins with an unasigned murmur3 hash

    • 回傳五篇最相關的 wiki 文章

      Feature hashing 的目標是把原始的高為特徵向量壓縮成較低為特徵向量，且勁量不損失原始特徵的表達。

  • Doc Reader

    • Paragraph encoding + Question encoding
    • 3-layer bidirectional LSTMs with h=128 hidden units for both
    • Stanford CoreNLP toolkit for tokenization, lemma, POS, NE tag
    • 所有訓練資料依照段落長度排序，並分到 mini-batches of 32 examples each.
    • 用 Adamax 優化
    • 在 word embeddings 和所有 hidden units of LSTMs 使用 dropout with p=0.3

    • Paragraph encoding
      $$ {p_1,…,p_m}=RNN({\check{p}_1,…\check{p}_m}) $$
      $p_i$: 為每層的 hidden units 的並列(concatenation)，預期 encode 有用的 token $p_i$ context 資訊
      $\check{p}_1$: feature vector

      1. Word embeddings: 使用 pre-trained 300d Glove word embeddings，保持大部分但微調前 1000 於問題的常見字，像是 what, how, which，畢竟這對QA系統很重要
        $$ f_{emb}(p_i)=E(p_i) $$

      2. Exact match: 使用3個 simple binary features，表示 $p_i$ 是否可完整配對到問題中的字，完整、小寫、或lemma form。但是3用在哪不解？
        $$ f_{exact\_match(p_i)}=I(p_i\in q) $$

      3. Token features: 再加上 POS, NER, TF(normalized term frequency)
        $$ f_{token}(p_i)=(POS(p_i),NER(p_i),TF(p_i))$$

      4. (複雜)Aligned question embedding: attention score $a_{i,j}$ captures the imilarity between $p_i$ and each question words $q_j$.
        $$ f_{align}(p_i)=\sum_j a_{i,j}E(q_j) $$

    • Question encoding

      • 套用在 $q_i$ word embedding
      • 並且結合 hidden units into one vector

    • Prediction

      • 使用 paragraph vectors 和 question vector 來訓練兩個 classifier
      • 用來預測答案的開頭字跟結尾字
      • 使用 bilinear term 計算每個字可能為開頭和結尾的機率
      • 選擇最可能的 span 從 $i$ 到 $i’$，不可超過十五個單字，且 $P_{start}(i)*P_{end}(i’)$ is max
      • 使用 unnormalized exp 以相容不同段落之間的比較
      • argmax 找出最可能的答案

Data

  1. Wiki: 答案來源

    • 只使用內容純文字，list, figure 等全部刪除
    • 5,075,182 articles, 9,008,962 unique uncased tokens

  2. SQuAD datset: 主要用來 train Document Reader

    • Stanford Question Answering Dataset, for machine comprehension based on Wiki
    • 87k examples for training; 10k for development, a large hidden test set(private)
    • Each example: a paragraph & answer, an associated human-generated question
    • 訓練過程中，需要預測出正確解答
    • 使用 exact string match(EM) 和 F1 score 評估，token level
    • 只使用 development set QA pairs 做評估，但不使用提供的段落，要求系統找出正確解答

  3. 另外 3 個 QA datasets: test 整個系統，evaluated 使用 multitask learning and distant supervision 學習的 model

    • CuratedTREC
      • from TREC QA tasks
      • 2,180 questions
    • WebQuestions
      • 被用來回答 Freebase KB 的問題
      • 問題是透過 Google Suggest API 來爬取的
      • 透過 Amazon Turk 得到解答
      • 使用 entity names 把答案轉換成 text
    • WikiMovies
      • 96k qa pair in movie
      • from OMDb and MovieLens DB
  • Distantly Supervised Data
    • 原本資料只有 qa pair，不含相關段落
    • 利用 Distant supervision for relation extraction(2009) 提出的方法
    1. 用 Doc. Retriever 找出該問題的前五篇
    2. 移除不包含答案、短於25或超過1500 char 的段落
    3. 如果 Q 裡面有 named entity，移除那些不包含 EN 的段落
    4. score all positions that match an answer using unigram and bigram overlap between the question and a 20 token window利用評分剩下的段落
    5. 留最高分的五個段落(keeping up to the top 5 paragraphs with the highest overlaps)，若都沒有，則移除該例子

Experiments

  • Document Retrieval

    • 方法為找出的前五篇文章中至少一篇出現答案的比率
    • 比 wiki 內建的搜尋引擎好
    • 比 Okapi BM25 或 cosine distance in the word embeddings space 好

  • Document Reader

    • 用 SQuAD evaluation 評估
    • 此 model 可以達到 SQuAD 最好的結果
    • 若拿掉 $f_{aligned}$ and $f_{exact\_match}$，效果大幅降低，推論出這兩個 features 扮演著相似但是互補的角色

  • DrQA

    • 使用上述四個 dataset

    • 用三個 DrQA 版本來比較餵給 Doc. Reader 的訓練來源上使用 distant supervision 和 multitask learning 的影響

      1. SQuAD: 單純在 Doc. Reader model 上使用訓練集和評估集
      2. Fine-tune (DS): Doc. Reader model 已經用 SQuAD 事先訓練好，再使用從每個 dataset 得到的 distant supervision training set 來 fine-tuned
      3. Multitask (DS): Doc. Reader model 從 SQuAD training set 和所有 DS sources 訓練出來

    • SQuAD 雖然比較差，但沒有落後很多，代表有 task transfer

    • SQuAD -> Multitask(DS) 最主要的進步原因不是 task transfer，因為 fine-tuning on each dataset alone using DS 也有進步，代表導入同領域的 extra data 提供幫助
    • Multitask (DS) system 最好
    • 比較 YodaQA(來源不只 wiki，redudant resources)，儘管我們的任務困難度上升，但重新確認我們的效能並不是落後太多 on CuratedTREC
    • on WebQuestions 落後較多是因為這個資料集是從特定格式的 Freebase 產生的，而 YodaQA 直接使用此資料
    • 完整的 single SQuDA 系統雖然在 SQuDA 資料上的成績從69.6 -> 27.1，但是因為給予的資料從段落變成所有 wiki，若給正確的該篇文章（非段落）則可達 49.4，表示 false positives 來自 topcial sentences.(?)
    • 儘管 Doc. Retrieval 效果很好，但是值得注意的是，這個落差可能來自於 SQuAD 的questions，因為這些questions是再給予一段段落後去產生的，並不知道上下文，因此可能和整篇文章的主旨有出入

Conclusion

  • 利用 wiki 研究 MRS
  • 此成果表示 MRS 是很關鍵的任務
  • 只使用 machine comprehension 並不能解決整體任務，此研究結合 search, distant supervision, and multitask learning
